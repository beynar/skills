# Sounds

We use sound to communicate information and enhance how merchants experience the Shopify admin. Sound patterns make interactions easier and more predictable.

## Principles

### Selectively Urgent

The duration, volume, and character of a sound should match the event urgency level.

**Urgency examples:**
- **High urgency:** Customer struggling with checkout requires immediate attention (more urgent sound)
- **Low urgency:** Customer arrived at storefront is informative, doesn't require immediate action (subtle sound)

**Do:**
- Match sound intensity to urgency
- Use duration appropriately
- Vary volume by urgency
- Customize for context

**Don't:**
- Use urgent sounds for minor events
- Create urgency where none exists

### Avoid Annoying Repetition

Some sounds occur many times per day. Appealing on first listen may become irritating after many repetitions.

**Key principle:** Sounds should be informative, not annoying.

**Strategy:**
- Use data to determine trigger frequency
- Test repetition tolerance
- Create sounds that age well
- Avoid overly distinctive or novel sounds

**Do:**
- Test repeated listening
- Consider daily frequency
- Keep simple and non-distracting

**Don't:**
- Create sounds that become annoying
- Use distinctive hooks that tire quickly
- Ignore repetition frequency

### Distinct and Succinct

Merchants hear sounds in various contexts (offices, homes, mobile devices).

**Sound characteristics:**
- Clean and focused
- Succinct and brief
- Convey information clearly
- Avoid muffled or distorted sounds

**Do:**
- Keep sounds clear
- Make concise sounds
- Ensure clarity across contexts
- Test in various environments

**Don't:**
- Create muffled or distorted audio
- Make overly long sounds
- Lose clarity through compression

### Test Across Devices

Speaker quality significantly affects sound perception.

**Device considerations:**
- Headphones reveal full frequency spectrum
- Loudspeakers provide good representation
- Smartphone speakers distort easily
- Lower volumes prevent distortion

**Testing approach:**
- Test on headphones
- Test on desktop speakers
- Test on mobile device speakers
- Test at multiple volume levels

**Do:**
- Test on variety of devices
- Test at different volumes
- Account for smaller speakers
- Create mobile-friendly versions

**Don't:**
- Test only on quality equipment
- Ignore mobile speaker limitations
- Assume one version works everywhere

### Think Beyond Sound

Not all merchants can rely on sound for cues.

**Accessibility considerations:**
- Hearing disabilities
- Auditory processing issues
- Merchants preferring silence
- Alternative communication methods

**Approach:**
- Always provide alternative feedback (visual, haptic)
- Don't assume sound perception
- Support preference for silence
- Include text-based notifications

**Do:**
- Provide visual alternatives
- Offer haptic feedback options
- Respect merchant preferences
- Never require sound-only feedback

**Don't:**
- Rely solely on sound
- Assume all merchants hear sounds
- Ignore accessibility needs

## When to Use Sound

Useful and well-planned sounds enhance merchant understanding. Unnecessary sounds reduce perceived quality.

**Key principle:** Merchants should always have the option to turn off sound.

## Common Sound Events

### Alerts

For events requiring immediate merchant attention.

**Purpose:**
- Encourage pause in workflow
- Signal urgent attention needed
- Create sense of importance

**Examples:**
- Critical system alerts
- High-priority issues
- Immediate action needed

### Notifications

For useful but non-urgent events.

**Purpose:**
- Informative feedback
- Not immediately necessary
- Can wait for merchant attention

**Examples:**
- Task completion
- New message arrival
- Status updates

### Feedback for Successful Action

Confirm successful completion.

**Purpose:**
- Reassure merchant
- Confirm action received
- Positive reinforcement

**Examples:**
- Save completed
- Upload finished
- Item purchased

### Feedback for Unsuccessful Action

Statement of failure requiring acknowledgment.

**Purpose:**
- Communicate problem
- Signal need for action
- Provide clarity

**Examples:**
- Validation error
- Upload failed
- Connection lost

## How to Approach Sound Design

Be creative and intentional. Remember:

### Duration Guidelines

- **Short sounds:** Maximum 250ms
- **Long sounds:** Maximum 1 second
- Keep sounds brief and punchy
- Test timing feels natural

### Sound Family Consistency

Sounds should feel part of same family.

**Do:**
- Maintain consistent tone/key
- Use similar timbre
- Build cohesive audio identity
- Ensure sounds work together

**Don't:**
- Introduce new notes without reason
- Mix conflicting musical styles
- Create disjointed audio

### Create Multiple Versions

Design two versions for each sound:

1. **Louder version** for desktop applications
2. **Quieter version** for mobile where speakers are smaller and distort easily

**Purpose:**
- Account for different speaker capabilities
- Prevent mobile distortion
- Maintain quality across platforms
- Optimize for device context

### Design with Intentionality

Sound design deserves same care as visual design.

**Do:**
- Plan sound strategically
- Consider context carefully
- Test thoroughly
- Iterate based on feedback

**Don't:**
- Add sounds as afterthought
- Use generic stock sounds
- Skip testing
- Ignore merchant preferences

## Implementation Strategy

1. Identify sound events needed
2. Determine urgency level
3. Create sound concepts
4. Design for family consistency
5. Create desktop and mobile versions
6. Test on multiple devices
7. Verify timing and duration
8. Get feedback
9. Implement with alternative feedback
10. Monitor usage and adjust
